# Tree Inventory Extension – Quality, Training & Change Management Plan

## 1. جودة الكود والاختبارات

### 1.1 إستراتيجية الاختبار
- **اختبارات الوحدة (Backend)**  
  - `TreeInventoryService`: سيناريوهات الزيادة، الفقد، التعديل، الحصاد مع التزامن.  
  - Serializers: التحقق من الحقول الإلزامية، الرسائل، وقواعد التحويل.  
  - Signals/Views: التأكد من استدعاء الخدمة في حالات الأنشطة المعنية فقط.
- **اختبارات التكامل (API)**  
  - سيناريو تسجيل نشاط -> إنشاء حدث مخزون وتحديث الرصيد.  
  - استرجاع `summary` بـ filters مختلفة والتحقق من الاستجابة.  
  - `events` endpoint مع التصفية والتقسيم.  
  - `advanced_report` مع خيار `include_tree_inventory`.
- **اختبارات الواجهة (Frontend)**  
  - `DailyLog`: ظهور/اختفاء الحقول الشرطية، التحقق من إلزامية الإدخال قبل الإرسال.  
  - التقارير الجديدة: تحميل البيانات، التصفية، التصدير.  
  - الأعمدة الاختيارية في التقارير الحالية.
- **اختبارات الأداء**  
  - قياس زمن استجابة `summary` و `events` على بيانات كبيرة.  
  - مراقبة عدد الاستعلامات لكل تقرير لضمان الكفاءة.
- **اختبارات الموثوقية**  
  - محاكاة فشل الخدمة أثناء التحديث والتحقق من التراجع السليم.  
  - إعادة محاولة (retry) التسجيل في حالة حدوث تعارض (race condition).

### 1.2 أدوات الاختبار
- Backend: `pytest`, `pytest-django`, `pytest-factoryboy`, `pytest-cov`.
- API: `rest_framework.test.APIClient`, سيناريوهات محددة عبر Fixtures.
- Frontend: `Vitest`, `@testing-library/react`, `MSW` لمحاكاة API.
- Performance: `pytest-benchmark` أو `Locust` لجولات تحميل مركّزة.

### 1.3 معايير التغطية
- لا يقل عن 85% تغطية للكود الجديد (خاصة الخدمات والـ API).  
- تغطية 100% للحالات الحرجة (التزامن، التحقق).

### 1.4 CI/CD
- تحديث خطوط CI لتشغيل:
  1. الهجرات + التحقق من مخطط قاعدة البيانات.  
  2. `pytest` مع تغطية.  
  3. `npm run test` للواجهة.  
  4. فحوصات lint (`flake8`, `black`, `eslint`, `prettier`).  
  5. نشر إلى بيئة Staging مع بيانات تجريبية.  
  6. إجراء اختبار قبول semi-automated قبل الإطلاق.

## 2. المراقبة والتشخيص
- إضافة تسجيل زمني في `TreeInventoryService` (عدد الأشجار قبل/بعد، المستخدم).  
- لوحات مراقبة (Grafana أو Django Admin charts):  
  - رصيد الأشجار per farm/location.  
  - أسباب الفقد الأكثر شيوعاً.  
  - استهلاك الماء/السماد مقارنة بالإنتاجية.  
- تنبيهات (Alerting) عند انخفاض الرصيد دون حد آمن أو عند زيادة الفقد في فترة قصيرة.

## 3. إدارة البيانات والترحيل
- **جمع البيانات الأولية**: العمل مع الفريق الميداني لإدخال أرصدة البداية (ملفات CSV أو نماذج).  
- **سكربت تحميل**: استخدام أمر Django `load_tree_inventory` لملء `LocationTreeStock` و`TreeStockEvent` بناءً على البيانات.  
- **التنظيف**: تقارير اكتشاف بيانات غير مكتملة (تيار بدون مصدر، رصيد سلبي).  
- **مهام دورية**:  
  - Job شهري لتحديث حالة الإنتاجية بناءً على العمر والإنتاج.  
  - Job أسبوعي للتحقق من التناقضات (رصيد سلبي، أحداث متداخلة).

## 4. إدارة التغيير والتدريب
- **المستهدفون**:  
  - المشرفون الميدانيون.  
  - محللو البيانات الزراعية.  
  - فريق الدعم الفني.
- **مواد التدريب**:  
  - دليل خطوة بخطوة لإدخال نشاط يتعلق بالأشجار (PDF + فيديو قصير).  
  - دليل التقارير: كيفية استنتاج المشاكل الزراعية من البيانات.  
  - لوحة مصطلحات (عربي/إنجليزي) لجميع الحقول.
- **جلسات التدريب**:  
  - ورشة تعريفية (حضورية أو عبر الفيديو).  
  - جلسة أسئلة مفتوحة بعد أسبوع من الإطلاق لجمع التغذية الراجعة.
- **قنوات الدعم**:  
  - قناة تواصل سريعة (Slack/Teams) مخصصة لفترة Hypercare (أول شهر).  
  - تصعيد المشكلات عبر نظام التذاكر مع اتفاقية مستوى خدمة (SLA).

## 5. خطة الإطلاق
- **Staging**:  
  - نشر التعديلات وتشغيل السكربت، اختبار البيانات التجريبية، إجراء اختبارات قبول.  
  - الموافقة من مدير النظام الزراعي.
- **Production**:  
  - تحديد نافذة صيانة (Maintenance Window).  
  - تشغيل السكربت، تحميل البيانات الأولية، تنفيذ Smoke Tests.  
  - مراقبة دقيقة للأنشطة خلال الأسبوع الأول.

## 6. المتابعة والتحسين المستمر
- مراجعة شهرية تساعد على:  
  - تحليل الأداء الزراعي بناءً على البيانات الجديدة.  
  - رصد مخاطر البيانات (نقص التحديث، إدخالات خاطئة).  
  - تحديد فرص مزيد من الأتمتة أو المؤشرات المتقدمة (Machine Learning).
