# دليل الانضمام السريع للمطورين الجدد

## 1. تصور عام للمنصة
- **خدمات خلفية مبنية على Django/DRF** مع تطبيقات منفصلة للحسابات، النواة، والتكاملات، إلى جانب دعم CORS، CSP، وJWT افتراضي لجميع الطلبات.【F:backend/smart_agri/settings.py†L18-L140】
- **التوجيه العام** يضع كل نقاط REST تحت `/api/` مع إصدار v1، ويوفر وثائق OpenAPI ونقاط صحة، إضافة إلى مجموعة صفحات HTMX مخدومة من الخادم للمهام الإدارية السريعة.【F:backend/smart_agri/urls.py†L22-L57】【F:backend/README_HTMX.md†L1-L4】
- **الهوية واللغة** مضبوطة على العربية ومنطقة آسيا/عدن، ما يسهّل توافق التواريخ والواجهات مع المستخدمين المحليين.【F:backend/smart_agri/settings.py†L82-L85】

## 2. طبقة البيانات ونطاق الصلاحيات
- تعتمد النماذج على طبقة حذف ناعم مشتركة (`SoftDeleteModel`) وتغطي كيانات المزارع، المواقع، الأصول، المحاصيل، الأنشطة اليومية، المخزون، والأصناف المرتبطة بالأشجار والمياه.【F:backend/smart_agri/core/models.py†L5-L222】
- تربط عضوية المزرعة المستخدمين بالأدوار (Admin/Manager/Supervisor/DataEntry/Viewer) مع فرض فريد لكل مستخدم-مزرعة، ما يشكل الأساس للـRBAC داخل المنصة.【F:backend/smart_agri/accounts/models.py†L6-L13】
- تفحص صلاحيات API عبر `FarmScopedPermission`، وتستخلص معرفات المزارع المسموح بها للمستخدم، وتمنع الوصول للموارد خارج نطاقه، مع دعم صلاحيات Django التقليدية عند توفرها.【F:backend/smart_agri/core/api.py†L59-L133】

## 3. إعداد البيئة وتشغيلها بسرعة
1. ثبّت Python 3.11 وPostgreSQL 16، ثم أنشئ قاعدة البيانات والمستخدم كما هو موضح في التعليمات الرسمية.【F:README_Install.md†L17-L24】
2. أنشئ بيئة افتراضية داخل مجلد `backend/`، ثبّت الحزم، انسخ ملف البيئة `.env`، ثم استورد قواعد بيانات الأشجار والباتشات المطلوبة قبل تشغيل الترحيلات وإنشاء superuser.【F:README_Install.md†L25-L38】
3. بعد تحميل الـfixtures، شغّل الخادم عبر `manage.py runserver` وتحقق من `/api/health` و`/api/docs` للتأكد من نجاح الإعداد.【F:README_Install.md†L41-L45】
4. لعوامل التشغيل على Windows/IIS توجد سكربتات Waitress ومهام إقلاع آلية يجب مراجعتها عند التحضير للإنتاج.【F:README_Install.md†L46-L67】

## 4. مكونات الواجهة الأمامية
- يقوم `app.jsx` بتحميل الصفحات كسولياً، يحوطها بـ`PrivateRoute`، ويقرر عرض الصفحات تبعاً لصلاحيات المستخدم (مثلاً المزارع، المحاصيل، التدقيق، الإعدادات)، مع واجهة عربية وموفر للتوست ومراقبة لحالة الاتصال.【F:frontend/src/app.jsx†L2-L112】
- تعتمد الواجهة على مزوّد قائمة انتظار غير متصلة (`OfflineQueueProvider`) لعرض حالة المزامنة، العدادات، والتحكم في محاولات التفريغ مع إشعارات للمستخدم.【F:frontend/src/offline/OfflineQueueProvider.jsx†L1-L136】
- يحتوي عميل API على منطق شامل لتحديد قاعدة العنوان، تخزين الرموز، توليد مفاتيح idempotency، إدارة الطوابير لكل مستخدم، وإعادة رفع المرفقات عند عودة الاتصال، مع سياسات backoff وفصل للطلبات اليومية عن الطلبات العامة.【F:frontend/src/api/client.js†L1-L680】
- يبرهن اختبار Vitest على حفظ سجلات اليوميات والمرفقات أثناء عدم الاتصال، والتفريغ الصحيح للطوابير، ما يوفر شبكة أمان لأي تغييرات مستقبلية على هذه الوظائف.【F:frontend/tests/offlineQueue.test.js†L1-L78】

## 5. دعم وضع عدم الاتصال وPWA
- يستخدم العميل IndexedDB (عبر `idb-keyval`) لتخزين الطلبات والمرفقات مع إعادة المحاولة وتتبّع المحاولات الفاشلة؛ كما يُرسل أحداثاً لتحديث الواجهة وإطلاق مزامنة خلفية عند الحاجة.【F:frontend/src/api/client.js†L52-L262】【F:frontend/src/api/client.js†L516-L680】
- يعمل Service Worker بسياسة cache-first للموارد الأساسية وصفحة احتياطية عند فقد الاتصال، ما يضمن تجربة PWA متماسكة في الحقول أو البيئات ذات الاتصال المحدود.【F:service-worker.js†L2-L10】

## 6. التكاملات والخدمات الخارجية
- توفر وحدة التكاملات دالة `dispatch_event` لإرسال Webhooks بتوقيع HMAC، مع إنشاء سجل تسليم وتشغيل الإرسال في خيوط منفصلة لضمان عدم حجب طلبات API الأساسية عند فشل الوجهة البعيدة.【F:backend/smart_agri/integrations/utils.py†L7-L30】

## 7. بوابة HTMX وواجهات الخادم
- إلى جانب واجهات React، يوجد مسار `/ui/*` لمشاهد HTMX الخفيفة (قوائم المزارع، إنشاء السجل اليومي، التدقيق، مهام المحاصيل) التي تسهّل إنجاز العمليات السريعة أو دعم المشغلين ذوي الأجهزة البسيطة.【F:backend/smart_agri/urls.py†L39-L57】【F:backend/README_HTMX.md†L1-L4】

## 8. ما يجب التركيز عليه في الأسابيع الأولى
- **التحكم في نطاق البيانات**: راجع كيفية تمرير `farm_id` إلى الاستعلامات واستخدام المساعدات مثل `_ensure_user_has_farm_access` و`_limit_queryset_to_user_farms` قبل بناء أي منطق جديد.【F:backend/smart_agri/core/api.py†L97-L133】
- **سير عمل السجلات اليومية**: افهم الربط بين `DailyLog` و`Activity` والمرفقات لضمان إدخال بيانات متكاملة عند تطوير شاشات أو عمليات جديدة للحقول.【F:backend/smart_agri/core/models.py†L150-L218】
- **المصادقة ودورة حياة الرموز**: تأكد من التعامل مع تحديث الرموز وحالة عدم الاتصال في `client.js` قبل إضافة طلبات جديدة إلى الواجهة الأمامية.【F:frontend/src/api/client.js†L341-L680】

## 9. خارطة تعلم مقترحة لاحقًا
1. **تعميق فهمك لـ Django REST Framework** (Serializers, ViewSets, فلاتر متقدمة) لتوسيع الموارد الحالية أو إضافة نسخ إصدار جديدة.【F:backend/smart_agri/settings.py†L93-L112】【F:backend/smart_agri/core/api.py†L10-L198】
2. **إدارة العضويات والأدوار** عبر دراسة العلاقة بين `FarmMembership` و`FarmScopedPermission` وكيفية الاستفادة من أذونات Django الأصلية عند الحاجة إلى حوكمة أدق.【F:backend/smart_agri/accounts/models.py†L6-L13】【F:backend/smart_agri/core/api.py†L59-L94】
3. **تقنيات PWA وقوائم الانتظار** لمواءمة تجربة العمل الميداني، بما في ذلك كيفية رفع المرفقات المخزنة وإعادة محاولات الطلبات من خلال `OfflineQueueProvider` والاختبارات المصاحبة.【F:frontend/src/offline/OfflineQueueProvider.jsx†L1-L136】【F:frontend/src/api/client.js†L516-L714】【F:frontend/tests/offlineQueue.test.js†L52-L77】
4. **التكاملات الخارجية ومراقبة الويب هوكس** لضبط التواقيع ومتابعة حالات الفشل مع backoff مناسب عند إضافة قنوات جديدة.【F:backend/smart_agri/integrations/utils.py†L7-L30】

باتباع هذا الدليل سيحصل المطوّر الجديد على صورة شاملة للمنصة، ويستطيع الانتقال سريعاً من الإعداد إلى المساهمة الفعالة مع إدراك للمناطق الحرجة التي تتطلب عناية خاصة.
